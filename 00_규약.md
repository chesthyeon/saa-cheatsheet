# SAA-C03 치트시트 작성 규약

> 이 문서는 모든 챕터 작성 시 일관성을 보장하기 위한 작성 규약입니다.
> Claude에게 챕터 작성을 요청할 때 항상 이 파일을 참조시킵니다.

## 1. 목적

- AWS SAA-C03 시험 합격 (1차 목표)
- Cloud/DevOps/SRE 직무 전환을 위한 AWS 서비스 이해도 확보 (2차 목표)
- dump 풀이의 "근거 있는 정답" 도출을 위한 키워드-서비스 매핑 사전 구축

## 2. 작성 원칙

1. **시험 우선, 깊이는 최소한**: 운영 디테일보다 "어떤 키워드 → 어떤 서비스"가 핵심
2. **표 + 키워드 + 함정 3종 세트**: 모든 챕터는 이 구조를 기본으로 함
3. **검증 가능성**: outdated 가능 정보(가격, 한도, 신규 기능)는 별도 섹션에 명시
4. **dump 연결성**: 챕터 끝에 "이 챕터로 풀 수 있는 dump 문제 유형" 제시

## 3. 챕터별 표준 구조

각 챕터 .md 파일은 다음 섹션을 순서대로 포함합니다:

```
# Chapter X-Y. [제목]

## 0. 한 줄 요약
(이 챕터에서 외워야 할 단 하나의 문장)

## 1. 핵심 서비스 표
| 서비스 | 한 줄 정의 | 주요 키워드 | 가격 모델 |

## 2. 키워드 → 서비스 매핑
(시험에서 이 단어가 보이면 → 이 서비스)

## 3. 헷갈리는 포인트 / 함정 (최소 3개)
- 함정 1: ...
- 함정 2: ...
- 함정 3: ...

## 4. 단골 시나리오 (해당 시)
(이 서비스가 등장하는 전형적인 아키텍처 패턴)

## 5. 검증 필요 항목 ⚠️
(가격, 한도, 신규 기능 등 Claude의 학습 데이터가 outdated일 수 있는 항목)
- [ ] AWS 공식 문서 링크: ...
- [ ] 확인 필요: ...

## 6. 이 챕터로 풀 수 있는 dump 유형
- 유형 1: "...라는 키워드가 나오면..."
- 유형 2: ...

## 7. 학습 메모 (Banghyeon 작성 영역)
(dump 풀면서 추가로 깨달은 점, 자주 틀리는 부분 등)
```

## 4. 포맷 규칙

- **언어**: 한국어 본문 + AWS 서비스명/기술 용어는 영어 원문 유지 (예: "Aurora Serverless v2", "Multi-AZ")
- **표기**: 한/영 병기는 첫 등장 시에만 (예: "교차 영역(Cross-Region)")
- **분량**: 챕터 1개당 .md 본문 기준 300~600줄 (너무 길면 분할)
- **표 사용**: 비교/매핑은 무조건 표, 설명은 짧은 단락 + bullet
- **코드블록**: CLI 명령어, JSON 정책, ARN 예시는 ` ```bash`, ` ```json`으로 명시
- **이모지**: ⚠️(주의), ✅(정답 패턴), ❌(오답 패턴), 💡(팁), 🔑(핵심 키워드)만 제한적 사용
- **링크**: AWS 공식 문서 링크는 "검증 필요 항목" 섹션에만 모아서 표기

## 5. Claude 요청 표준 프롬프트

챕터 작성 요청 시 다음 형식으로 보냅니다:

```
"Chapter [X-Y] [제목] 치트시트 작성해줘.
- 작성 규약: 00_규약.md 기준
- 포함 서비스: [...]
- 출력: .md 형식, 표준 7개 섹션 모두 포함
- 검증 필요 항목 섹션은 반드시 별도로 명시"
```

## 6. 챕터 작성 워크플로우

> 매 챕터 작성 시 아래 순서를 반드시 따릅니다.

### 6-1. 작성 파이프라인 (Claude 실행)

1. **`.md` 파일 작성** → 해당 챕터 폴더에 저장 (표준 7개 섹션 구조)
2. **Git commit + push** → 원격 저장소에 반영
3. **README.md 업데이트** → 챕터 진행 상태 표 갱신

### 6-2. dump 검증 (Banghyeon 실행)

1. 해당 영역 dump 문제 5~10개 풀이
2. 틀린 문제 → 해당 챕터 .md의 "학습 메모" 섹션에 약점 추가 기록
3. 재학습 필요 시 → Claude에게 "Chapter X-Y에서 [구체적 약점] 보강해줘" 요청

## 7. Git 커밋 메시지 규칙

- `feat(ch1-1): 컴퓨팅 챕터 초안 작성`
- `fix(ch1-1): Fargate 가격 모델 정정 (dump #47에서 틀림)`
- `docs(ch1-1): 학습 메모 추가 - Lambda 동시성 한도`
- `refactor(ch3): EC2 vs Lambda 비교표 재구성`

`fix:` 커밋만 모아 보면 본인의 약점 영역이 한눈에 드러납니다.

## 8. 금지 사항

- ❌ 챕터를 한 번에 다 작성하려 하지 말 것 (퀄리티 저하)
- ❌ 자작 모의 문제 만들지 말 것 (자기 이해 수준을 못 넘음, dump가 더 정확)
- ❌ "검증 필요 항목"을 비워두지 말 것 (Claude의 정보는 outdated 가능)

## 9. 챕터 작성 권장 순서

목차 번호 순서가 아닌, **체득 효율 순서**로 작성합니다:

1. Chapter 1 (키워드 매핑) — 전체 그림
2. Chapter 3 (헷갈리는 쌍) — 1을 보강
3. Chapter 4 (시나리오 템플릿) — 1+3 응용
4. Chapter 5 (함정) — 4까지 본 뒤 정리
5. Chapter 2 (목적별 패턴) — 1~4의 재구성
6. Chapter 0 (Well-Architected) — 시험 직전 정독용
