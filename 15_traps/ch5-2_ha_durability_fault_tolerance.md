# Chapter 5-2. 고가용성 vs 내구성 vs 내결함성

## 0. 한 줄 요약

🔑 **"High Availability = 서비스 중단 최소화, Durability = 데이터 손실 방지, Fault Tolerance = 장애가 나도 성능 저하 없이 계속 작동" — 세 단어의 정확한 구분이 정답을 바꾼다**

---

## 1. 핵심 정의 비교

| 개념 | 정의 | 핵심 질문 | 측정 |
|------|------|----------|------|
| **High Availability (HA)** | 시스템이 **가동 상태를 유지**하는 능력 | "서비스 접근 가능한가?" | **Uptime %** (99.9%, 99.99%) |
| **Durability** | 데이터가 **손실 없이 보존**되는 능력 | "데이터가 안전한가?" | **Nines** (99.999999999%) |
| **Fault Tolerance (FT)** | 장애 발생 시에도 **성능 저하 없이** 작동 | "장애 중에도 정상 작동하는가?" | **무중단 운영** |

### 비유로 이해

| 개념 | 비유 |
|------|------|
| **HA** | 식당이 하루에 1분만 문 닫음 (99.99% 영업) |
| **Durability** | 식당 레시피가 절대 분실되지 않음 |
| **FT** | 요리사 1명이 쓰러져도 다른 요리사가 즉시 대체, 주문 지연 없음 |

---

## 2. 관계도

```
Fault Tolerance (가장 강력)
  └─ 포함 → High Availability
                └─ 다른 축 → Durability (데이터 관점)

FT > HA: 내결함성이 확보되면 고가용성은 자동 달성
HA ≠ FT: HA는 "빠른 복구"를 허용, FT는 "복구 자체가 불필요"
HA ≠ Durability: 서비스가 다운되어도 데이터는 안전할 수 있음
```

---

## 3. AWS 서비스별 구현

### High Availability (HA) 구현

| 패턴 | 서비스 | 설명 |
|------|--------|------|
| **Multi-AZ** | RDS, ElastiCache, NAT GW | AZ 장애 시 자동 페일오버 (60~120초 중단) |
| **ASG** | EC2 | 인스턴스 장애 시 자동 교체 |
| **ALB** | 로드밸런서 | 비정상 인스턴스 제거, 정상만 라우팅 |
| **Route 53 Failover** | DNS | 리전 장애 시 DR로 전환 |
| **S3** | 스토리지 | 99.99% 가용성 (Standard) |

💡 HA 핵심 = **"짧은 중단 후 복구"** 허용

### Durability 구현

| 서비스 | 내구성 | 설명 |
|--------|--------|------|
| **S3 Standard** | **99.999999999%** (11 nines) | 3개 AZ에 자동 복제 |
| **S3 Glacier** | 99.999999999% | 장기 아카이브 |
| **EBS** | 99.999% (5 nines) | AZ 내 복제 |
| **RDS Multi-AZ** | 동기 복제 | 데이터 손실 최소화 |
| **DynamoDB** | 99.999999999% | 3개 AZ 자동 복제 |

💡 Durability 핵심 = **"데이터 자체의 안전성"** (서비스 가동과 무관)

### Fault Tolerance 구현

| 패턴 | 서비스 | 설명 |
|------|--------|------|
| **Active-Active** | Multi-Site DR | 양쪽 모두 활성, 한쪽 장애 = 즉시 흡수 |
| **SQS** | 메시징 | 소비자 장애 시 메시지 보존, 복구 후 처리 |
| **Lambda** | 컴퓨팅 | 자동 재시도, 다중 AZ 실행 |
| **DynamoDB Global Tables** | DB | Active-Active, 리전 장애에도 무중단 |
| **Aurora Multi-Master** | DB | 모든 노드 쓰기 가능, 한 노드 장애 무영향 |

💡 FT 핵심 = **"장애가 발생해도 사용자가 모름"**

## 4. 시험 키워드 → 개념 매핑

| 시험 키워드 | 개념 | 정답 방향 |
|------------|------|----------|
| "최소 다운타임", "가용성 확보" | **HA** | Multi-AZ, ASG, ALB |
| "데이터 손실 방지", "내구성" | **Durability** | S3, Multi-AZ RDS, 백업 |
| "장애 시에도 무중단", "성능 저하 없이" | **FT** | Active-Active, 다중 리전 |
| "99.99% 가동 시간" | **HA** | SLA 기반 아키텍처 |
| "11 nines" | **Durability** | S3 |
| "자동 페일오버" | **HA** (짧은 중단 허용) | Multi-AZ RDS |
| "제로 다운타임" | **FT** | Active-Active |
| "RPO = 0" | **Durability** + FT | 동기 복제 |
| "RTO = 0" | **FT** | Active-Active |

## 5. 시험 빈출 시나리오

### 시나리오 1: "데이터 손실을 방지하면서 가용성 확보"

> Durability + HA

**정답**: RDS **Multi-AZ** (동기 복제 = 데이터 안전 + 자동 페일오버)  
**오답 함정**: Read Replica (비동기 = 데이터 손실 가능)

### 시나리오 2: "장애 시에도 성능 저하 없이 서비스"

> Fault Tolerance

**정답**: **Multi-Site Active-Active** + Aurora Global DB  
**오답 함정**: Multi-AZ (페일오버 중 60~120초 중단 = FT 아님)

### 시나리오 3: "99.99% 가용성 SLA"

> HA

**정답**: Multi-AZ + ALB + ASG (다중 AZ 인프라)  
**오답 함정**: 단일 AZ (AZ 장애 시 다운)

### 시나리오 4: "S3에 저장된 데이터의 내구성"

> Durability

**정답**: S3 Standard = **11 nines** (기본 제공, 추가 설정 불필요)  
**추가 보호**: S3 Versioning, Cross-Region Replication, Object Lock

### 시나리오 5: "EBS 볼륨 장애 대비"

> Durability

**정답**: **EBS 스냅샷** (S3에 저장 = 11 nines 내구성)  
**오답 함정**: EBS 자체 (5 nines, AZ 장애 시 손실 가능)

### 시나리오 6: HA vs FT 판별

| 문제 표현 | 정답 개념 |
|----------|----------|
| "최소 다운타임" | **HA** (짧은 중단 OK) |
| "다운타임 제로" | **FT** (중단 자체 불가) |
| "자동 복구" | **HA** |
| "복구 없이 계속 작동" | **FT** |

## 6. 헷갈리는 포인트 / 함정

### 함정 1: Multi-AZ RDS = HA이지 FT가 아니다

- Multi-AZ: 페일오버 시 **60~120초 중단** 발생
- 이것은 HA (빠른 복구)이지 FT (무중단)가 아님
- 💡 "제로 다운타임 DB" → Multi-AZ ❌ → **Aurora Multi-Master** 또는 **DynamoDB Global Tables**

### 함정 2: S3 가용성 vs 내구성

| 클래스 | 가용성 | 내구성 |
|--------|--------|--------|
| Standard | 99.99% | 99.999999999% |
| Standard-IA | 99.9% | 99.999999999% |
| One Zone-IA | 99.5% | 99.999999999% (단일 AZ) |
| Glacier | 99.99% | 99.999999999% |

- 💡 내구성은 모두 11 nines로 동일
- 💡 **가용성만 다름** (접근 가능 시간 차이)
- 💡 One Zone-IA: AZ 파괴 시 내구성도 상실 (유일한 예외)

### 함정 3: 가용성 숫자의 의미

| 가용성 | 연간 다운타임 |
|--------|-------------|
| 99% | 3.65일 |
| 99.9% | 8.77시간 |
| 99.99% | 52.6분 |
| 99.999% | 5.26분 |
