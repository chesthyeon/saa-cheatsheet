# Chapter 5-3. 보안 그룹(Stateful) vs NACL(Stateless)

## 0. 한 줄 요약

🔑 **"Security Group = 인스턴스 레벨 방화벽(Stateful, 허용만), NACL = 서브넷 레벨 방화벽(Stateless, 허용+거부)" — Stateful vs Stateless 차이가 시험의 핵심**

---

## 1. 전체 비교

| 기준 | Security Group (SG) | Network ACL (NACL) |
|------|--------------------|--------------------|
| **레벨** | **인스턴스** (ENI) | **서브넷** |
| **상태** | **Stateful** | **Stateless** |
| **규칙 유형** | **허용만** (Allow only) | **허용 + 거부** (Allow & Deny) |
| **기본값** | 모든 인바운드 거부, 아웃바운드 허용 | 모든 트래픽 허용 (기본 NACL) |
| **규칙 평가** | **모든 규칙 평가** (합집합) | **번호 순서대로** (첫 매칭 적용) |
| **적용** | 인스턴스에 명시적 연결 | 서브넷 내 모든 리소스 자동 적용 |
| **반환 트래픽** | **자동 허용** | **명시적 허용 필요** |
| **사용 빈도 (시험)** | ⭐⭐⭐ | ⭐⭐ |

---

## 2. Stateful vs Stateless 핵심

### Stateful (Security Group)

```
[요청] 인바운드 허용 (포트 80) → 자동으로 응답 아웃바운드도 허용
[응답] 아웃바운드 ← 별도 규칙 불필요 (상태 추적)
```

- 인바운드를 허용하면 → 해당 연결의 아웃바운드(응답)는 **자동 허용**
- 아웃바운드를 허용하면 → 해당 연결의 인바운드(응답)는 **자동 허용**
- 💡 규칙을 한 방향만 설정하면 됨

### Stateless (NACL)

```
[요청] 인바운드 허용 (포트 80) → 응답 아웃바운드는 별도 규칙 필요!
[응답] 아웃바운드 규칙: Ephemeral Port (1024-65535) 허용 필요
```

- 인바운드를 허용해도 → 아웃바운드(응답)는 **별도 허용 필요**
- 💡 **양방향 규칙 모두 설정** 해야 통신 가능
- 💡 응답 포트 = **Ephemeral Port** (1024-65535)

---

## 3. Ephemeral Port (임시 포트)

| OS | Ephemeral Port 범위 |
|----|---------------------|
| Linux | 32768 - 65535 |
| Windows | 49152 - 65535 |
| AWS 권장 | **1024 - 65535** (모두 커버) |

**NACL에서 반드시 허용:**
- 인바운드: 서비스 포트 (80, 443, 3306 등)
- 아웃바운드: **Ephemeral Port** (1024-65535) → 응답 허용

---

## 4. 규칙 평가 방식

### Security Group: 모든 규칙 합집합

```
규칙 1: 포트 80 허용 (HTTP)
규칙 2: 포트 443 허용 (HTTPS)
규칙 3: 포트 22 허용 (SSH, 10.0.0.0/8)

→ 세 규칙 모두 적용 (합집합)
→ 매칭 안 되면 = 묵시적 거부
```

### NACL: 번호 순서대로 (첫 매칭)

```
규칙 100: 포트 80 허용
규칙 200: 포트 443 허용
규칙 300: 10.0.0.5 거부 (특정 IP 차단)
규칙  *  : 모든 트래픽 거부 (기본)

→ 10.0.0.5에서 포트 80 요청 → 규칙 100에서 먼저 허용!
→ 특정 IP 차단하려면 규칙 300보다 낮은 번호 필요
```

💡 **NACL에서 특정 IP 차단**: 차단 규칙 번호 < 허용 규칙 번호

## 5. 기본값 비교

| 항목 | SG (새로 생성) | 기본 NACL | 커스텀 NACL |
|------|--------------|----------|------------|
| 인바운드 | **모두 거부** | 모두 허용 | **모두 거부** |
| 아웃바운드 | **모두 허용** | 모두 허용 | **모두 거부** |

💡 기본 NACL = 모두 허용 (VPC 생성 시 자동)  
💡 커스텀 NACL = **모두 거부** (주의!)

## 6. 시험 빈출 시나리오

### 시나리오 1: 특정 IP 차단

> "특정 IP 주소에서 오는 트래픽을 차단해야 한다."

**정답**: **NACL** (Deny 규칙 지원)  
**오답 함정**: Security Group (Deny 규칙 없음, 허용만 가능)  
💡 "차단", "Deny", "Block" → **NACL**

### 시나리오 2: 인스턴스 레벨 접근 제어

> "특정 EC2 인스턴스만 다른 인스턴스에 접근하도록."

**정답**: **Security Group** (SG ID로 참조)  
**오답 함정**: NACL (서브넷 레벨, 인스턴스 구분 불가)

### 시나리오 3: 웹 서버 접근 허용

> "웹 서버에 HTTP/HTTPS만 허용. 설정 최소화."

**정답**: **Security Group** (인바운드 80, 443 허용 → 응답 자동)  
**NACL로 하면**: 인바운드 80, 443 허용 + 아웃바운드 Ephemeral 허용 (복잡)

### 시나리오 4: NACL 설정 후 통신 안 됨

> "NACL에 인바운드 포트 80을 허용했지만 웹 접근이 안 된다."

**정답**: 아웃바운드에 **Ephemeral Port (1024-65535) 허용** 누락  
💡 NACL Stateless → 양방향 규칙 필수

### 시나리오 5: SG Chaining (참조)

> "ALB → EC2 → RDS 3-Tier에서 각 계층만 접근 허용."

**정답**: 보안 그룹 참조 (SG Chaining)
- EC2 SG: 인바운드 = ALB SG에서 80 허용
- RDS SG: 인바운드 = EC2 SG에서 3306 허용

💡 IP 대신 SG ID 참조 → 인스턴스 추가/제거 시 자동 적용

### 시나리오 6: 서브넷 레벨 방어

> "서브넷 전체에 대한 추가 방어 레이어가 필요하다."

**정답**: **NACL** (서브넷 레벨 = 서브넷 내 모든 리소스에 적용)  
💡 SG + NACL = **Defense in Depth** (심층 방어)

### 시나리오 7: DDoS 방어에서의 역할

```
방어 레이어:
  1. CloudFront + Shield (엣지)
  2. WAF (L7 필터링)
  3. NACL (L3/L4 IP/포트 차단)
  4. Security Group (인스턴스 레벨)
```

## 8. 헷갈리는 포인트 / 함정

### 함정 1: SG는 Deny를 할 수 없다

- SG: 허용 규칙만 가능 → 허용 안 된 것 = 묵시적 거부
- 특정 IP를 명시적으로 차단 → **불가능** → NACL 사용
- 💡 "Block", "Deny", "차단" → NACL

### 함정 2: NACL 규칙 번호 순서

- 낮은 번호가 먼저 평가
- 규칙 100 (허용) vs 규칙 200 (거부) → **100이 우선** (허용됨)
- 💡 차단이 우선되려면 차단 규칙 번호를 더 낮게 설정

### 함정 3: 기본 NACL vs 커스텀 NACL

- 기본 NACL: 모두 허용 → 방어 효과 없음
- **커스텀 NACL**: 모두 거부 → 규칙 추가 필수
- 💡 새 서브넷에 커스텀 NACL 연결 후 통신 안 됨 → 규칙 추가 누락

### 함정 4: SG는 여러 개 연결 가능

- 하나의 인스턴스에 **최대 5개 SG** 연결 가능
- 모든 SG의 규칙이 합쳐져서 평가 (합집합)
- 💡 "SG 여러 개" → 가장 넓은 허용이 적용됨
